# Backup and Restore

---

## 1. Backup Scope

Backups include all user-generated data and instance configuration. Canonical reference data (cards, sets, etc.) is excluded because it can be re-downloaded from update packages after a restore.

### Included tables

| Table | Contents |
|-------|----------|
| `users` | User accounts (username, role, auth type, state, password hash, OAuth fields) |
| `user_preferences` | Per-user preferences (default page, set completion mode) |
| `collection_entries` | Standard card collection records |
| `serialized_entries` | Serialized card tracker records |
| `slab_entries` | Slab tracker records |
| `sealed_inventory_entries` | Sealed product inventory records |
| `wishlist_entries` | Wishlist records |
| `app_settings` | Instance configuration (backup schedule, retention, schema version, etc.) |
| `user_export_files` | Removed-user export file metadata |
| `admin_notifications` | Admin notification records |
| `grading_agencies` | Both canonical and local grading agencies |
| `backup_records` | Backup history records |
| `backup_destination_records` | Per-destination backup attempt results |
| `backup_destination_configs` | Configured backup destination definitions |

### Excluded tables

| Table | Reason |
|-------|--------|
| `cards` | Canonical data - re-downloaded from update packages |
| `sets` | Canonical data - re-downloaded from update packages |
| `treatments` | Canonical data - re-downloaded from update packages |
| `sealed_products` | Canonical data - re-downloaded from update packages |
| `update_versions` | Canonical data provenance - rebuilt on next update |
| `pending_schema_updates` | Re-detected from manifest on next check |

After a restore, run a content update to repopulate canonical data.

## 2. Backup Types

Two backup types are stored independently with separate retention counts.

| Type | Label format | When triggered | Retention key | Default retention |
|------|-------------|----------------|---------------|-------------------|
| Scheduled | `{instanceName}-scheduled-{yyyyMMddHHmmss}` | Backup schedule (default weekly) or manual trigger | `backup_retention_scheduled` | 4 |
| Pre-update | `{instanceName}-pre-update-v{schemaVersion}-{yyyyMMddHHmmss}` | Automatically before any schema update | `backup_retention_pre_update` | 4 |

Retention counts are configured independently. Pruning occurs after each successful new backup of the same type. Records older than the retention count have `isAvailable` set to `false` and their files are deleted from all active destinations.

## 3. Backup Format

Each backup is a ZIP archive containing two files:

### `metadata.json`

```json
{
  "backupFormatVersion": 1,
  "instanceName": "my-instance",
  "schemaVersion": 1,
  "backupType": "scheduled",
  "timestamp": "2026-03-08T03:00:00Z",
  "label": "my-instance-scheduled-20260308030000"
}
```

| Field | Description |
|-------|-------------|
| `backupFormatVersion` | Format version for future compatibility checks |
| `instanceName` | Value of `INSTANCE_NAME` at backup time |
| `schemaVersion` | Schema version at backup time (critical for restore validation) |
| `backupType` | `"scheduled"` or `"pre-update"` |
| `timestamp` | UTC timestamp of when the backup was taken |
| `label` | File-system-safe label used as the ZIP filename |

### `dump.sql`

A PostgreSQL dump generated by `pg_dump` with `--no-owner --no-privileges`, scoped to only the included tables (via `--table=<name>` flags for each table in `BackupScope.Tables`).

## 4. Backup Destinations

All active backup destinations receive every backup. A destination failure does not block other destinations - each is attempted independently.

### Destination isolation

If a destination throws an exception during write:
1. The error is logged
2. An admin notification is created with the message `"Backup destination '{label}' failed: {error}"`
3. The backup continues to other destinations
4. The backup record is saved to the database regardless of destination outcomes

### Implemented destinations

| Type | Status |
|------|--------|
| `local` | Fully implemented - writes `.zip` files to a local filesystem path |
| `azure-blob` | Stub - not yet implemented (requires Azure.Storage.Blobs SDK) |
| `aws-s3` | Stub - not yet implemented (requires AWSSDK.S3 SDK) |
| `gcp-storage` | Stub - not yet implemented (requires Google.Cloud.Storage.V1 SDK) |

### Configuration

See [docs/configuration.md](configuration.md) for destination configuration fields and the destination management API.

## 5. Backup Schedule

| Value | Meaning |
|-------|---------|
| `weekly` (default) | Every Sunday; next run calculated at 03:00 UTC |
| `daily` | Every day at 03:00 UTC |

The schedule is stored in the `backup_schedule` app setting. Changed via `PATCH /api/settings/backup`. The backup container in Docker Compose also reads this value from environment variables.

Manual backups can be triggered at any time via `POST /api/backup/trigger`.

---

## 6. Retention Policy

Retention counts are stored independently for each backup type:

| Setting key | Default | Controls |
|-------------|---------|---------|
| `backup_retention_scheduled` | 4 | Number of scheduled backups to keep |
| `backup_retention_pre_update` | 4 | Number of pre-update backups to keep |

After each successful backup, the system queries for records of the same type ordered by creation time descending, skips the retention count, and marks the remainder as unavailable. File deletion is attempted on all active destinations for each pruned backup. Destination deletion failures are logged as warnings but do not block pruning from completing.

## 7. Restore Workflow

The same restore logic applies for all scenarios: recovering the same instance, migrating to a new instance or deployment type, and restoring during initial setup.

### Restore from the admin panel

`POST /api/restore` - upload a backup ZIP file (up to 500 MB). Admin authentication required.

`POST /api/restore/{backupId}` - restore from an existing backup record in local storage. Admin authentication required.

### Restore process

1. Read and parse `metadata.json` from the ZIP archive
2. Validate the schema version (see Section 8)
3. Read `dump.sql` from the archive
4. Drop all tables in `BackupScope.Tables` with `CASCADE`
5. Execute `dump.sql` via `psql` to restore the data
6. Return `{"restoredSchemaVersion": N}` on success

### Post-restore steps

After a restore, canonical data is absent. Take these steps:
1. Trigger a content update from the admin panel (`POST /api/updates/check`) to repopulate cards, sets, treatments, and sealed products
2. If the restored schema version is behind the current content version, a schema update may be required first

---

## 8. Schema Version Validation

The restore process checks whether the backup's schema version is compatible with the current deployment.

**Block condition:** If `backup.schemaVersion > deployment.applicationSchemaVersion`, the restore is blocked.

**Error response:** HTTP 409 Conflict with:
```json
{
  "error": "Cannot restore: backup schema version {N} is newer than current deployment schema version {M}. Update the deployment to schema version {N} or higher before restoring."
}
```

**Resolution:** Update the application to a version that includes schema version N or higher, then retry the restore.

---

## 9. Pre-Update Backups

Pre-update backups are taken automatically and silently before any schema update begins. They do not require admin confirmation.

- Label includes the target schema version: `{instanceName}-pre-update-v{schemaVersion}-{timestamp}`
- Failure blocks the schema update from proceeding
- Stored in all active destinations alongside scheduled backups
- Retention configured independently via `backup_retention_pre_update`

If a schema update fails after a successful pre-update backup, the system automatically attempts to restore from the pre-update backup. The admin is notified of the failure, the restore result, and any further recovery instructions needed.

---

## 10. Docker Startup Migration

When the application container starts, the `StartupMigrationService` (a hosted background service that runs before other services) checks for pending EF Core migrations.

**If migrations are pending:**
1. Take a pre-update backup silently
2. If backup fails: abort startup, create admin notification with recovery instructions, do not apply migrations
3. If backup succeeds: run `database.MigrateAsync()`
4. If migration fails: attempt automatic restore from the backup, abort startup, notify admin
5. If migration succeeds: continue startup normally

**If no migrations are pending:** Continue startup normally with no backup taken.

This behavior ensures data is protected before any schema change on Docker deployments where schema migrations may be included with application updates.

## 11. Admin Panel

The backup status endpoint provides a summary view for the admin panel:

`GET /api/backup/status` returns:
- Last scheduled backup (id, label, timestamp, schema version)
- Last pre-update backup (id, label, timestamp, schema version)
- Next scheduled backup time
- List of active backup destinations

`GET /api/backup/history` returns paginated backup history with type, schema version, file size, and availability flag.

`GET /api/backup/{id}/download` downloads the backup ZIP from local storage if available. Returns HTTP 410 if the backup has been pruned.

`POST /api/backup/destinations/{id}/test` tests connectivity to a destination and returns `{"success": true/false}`.

All backup and restore endpoints require Admin authentication.
