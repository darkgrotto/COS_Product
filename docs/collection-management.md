# Collection Management

---

## 1. Overview

CountOrSell tracks four distinct collection types. Each has its own data model, API endpoints, and display characteristics.

| Type | Quantity | Condition | Acquisition fields | Notes |
|------|---------|-----------|-------------------|-------|
| Standard card collection | Integer (multiple copies) | Yes | Date + price, both with defaults | Tracks regular copies |
| Serialized card tracker | 1 per record | Yes | Date + price, no defaults | Each record is a unique serial number |
| Slab tracker | 1 per record | Yes | Date + price, no defaults | Adds grading agency, grade, certificate |
| Sealed product inventory | Integer (multiple copies) | Yes | Date + price, both with defaults | Sealed product reference data required |

Admins have no collection. Only general users track collections.

---

## 2. Standard Card Collection

Each record in the standard collection represents one or more copies of a specific card in a specific treatment and condition.

| Field | Type | Required | Default | Notes |
|-------|------|---------|---------|-------|
| `cardIdentifier` | string | Yes | - | Pattern `^[a-z0-9]{3,4}[0-9]{3,4}$` (stored lowercase, displayed uppercase) |
| `treatment` | string | Yes | - | Key from the treatments reference table |
| `quantity` | integer | Yes | - | Minimum 1; adjusted via `PATCH /api/collection/{id}/quantity` |
| `condition` | enum | Yes | - | One of: `NM`, `LP`, `MP`, `HP`, `DMG` |
| `autographed` | boolean | Yes | `false` | When true, displayed as "NM - Autographed" etc. |
| `acquisitionDate` | date | Yes | Current date at time of entry | Editable after entry |
| `acquisitionPrice` | decimal | Yes | Current market value at time of entry | Editable after entry |
| `notes` | string | No | null | Free text |

The `cardIdentifier` is stored lowercase and displayed uppercase in all API responses.

---

## 3. Serialized Card Tracker

Each record represents a single serialized copy of a card. Acquisition date and price have no defaults - they must be entered at the time of addition.

| Field | Type | Required | Default | Notes |
|-------|------|---------|---------|-------|
| `cardIdentifier` | string | Yes | - | Same pattern as standard collection |
| `treatment` | string | Yes | - | Key from treatments reference table |
| `serialNumber` | integer | Yes | - | The card's individual serial number |
| `printRunTotal` | integer | Yes | - | Total print run size |
| `condition` | enum | Yes | - | `NM`, `LP`, `MP`, `HP`, or `DMG` |
| `autographed` | boolean | Yes | `false` | Boolean, separate from condition |
| `acquisitionDate` | date | Yes | No default | Must be entered at time of addition |
| `acquisitionPrice` | decimal | Yes | No default | Must be entered at time of addition |
| `notes` | string | No | null | Free text |

There is no quantity field - each serialized entry represents exactly one physical card.

---

## 4. Slab Tracker

Each record represents a single graded and slabbed card. Slabs may optionally reference a serial number if the slabbed card is also serialized.

| Field | Type | Required | Default | Notes |
|-------|------|---------|---------|-------|
| `cardIdentifier` | string | Yes | - | Same pattern as standard collection |
| `treatment` | string | Yes | - | Key from treatments reference table |
| `gradingAgency` | string | Yes | - | Agency code (e.g., `PSA`); validated against agency table |
| `grade` | string | Yes | - | Grade as assigned by the agency (e.g., `9`, `9.5`, `Gem Mint`) |
| `certificateNumber` | string | Yes | - | Certificate number assigned by grading agency |
| `serialNumber` | integer | No | null | Present only if the slabbed card is also serialized |
| `printRunTotal` | integer | Conditional | null | Required if `serialNumber` is provided; absent otherwise |
| `condition` | enum | Yes | - | `NM`, `LP`, `MP`, `HP`, or `DMG` |
| `autographed` | boolean | Yes | `false` | Boolean, separate from condition |
| `acquisitionDate` | date | Yes | No default | Must be entered at time of addition |
| `acquisitionPrice` | decimal | Yes | No default | Must be entered at time of addition |
| `notes` | string | No | null | Free text |

The API enforces that `printRunTotal` is required when `serialNumber` is provided. Providing `serialNumber` without `printRunTotal` returns HTTP 400.

### Certificate validation display

Each grading agency has a `validationUrlTemplate` and a `supportsDirectLookup` flag.

- **`supportsDirectLookup: true`** - A "Verify Certificate" link is generated by interpolating the certificate number into the URL template. Clicking it goes directly to the certificate record.
- **`supportsDirectLookup: false`** - A "Verify Certificate" link opens the agency's landing page. The certificate number is displayed prominently alongside the link for manual entry by the user. CCC uses this mode.

### Canonical grading agencies

These agencies are provided by update packages and cannot be modified by Product instance admins:

| Code | Full Name | Direct Lookup |
|------|-----------|--------------|
| BGS | Beckett Grading Services | Yes (URL TBD) |
| PSA | Professional Sports Authenticator | Yes (URL TBD) |
| SGC | Sportscard Guaranty | Yes (URL TBD) |
| CGC | Certified Guaranty Company | Yes (URL TBD) |
| CCC | Certificateur de Cartes de Collection | No (landing page only) |
| ISA | International Sports Authentication | Yes (URL TBD) |

Product instance admins can add local agencies via `POST /api/grading-agencies`. Local agencies can be edited and deleted; canonical agencies cannot.

**Local agency deletion with existing slab records:** The API returns `409 Conflict` with `{"requiresReplacement": true, "recordCount": N}`. The admin must provide a `replacementCode` in the request body; all affected slab records are remapped to the replacement agency before deletion.

**Agency code collision during updates:** If an update package introduces a canonical agency whose code matches an existing local agency, the admin is notified and must acknowledge before the update proceeds. All records referencing the local agency are remapped to the canonical version automatically.

## 5. Sealed Product Inventory

Each record represents one or more units of a sealed product. Sealed products cannot be marked as opened.

| Field | Type | Required | Default | Notes |
|-------|------|---------|---------|-------|
| `productIdentifier` | string | Yes | - | References the sealed product reference table |
| `quantity` | integer | Yes | - | Minimum 1 |
| `condition` | enum | Yes | - | `NM`, `LP`, `MP`, `HP`, or `DMG` |
| `acquisitionDate` | date | Yes | Current date at time of entry | Editable after entry |
| `acquisitionPrice` | decimal | Yes | Current market value at time of entry | Editable after entry |
| `notes` | string | No | null | Free text |

There is no "open" action. Sealed product is tracked as an inventory item only.

## 6. Card Identifiers

Card identifiers uniquely identify a specific card within a set.

**Pattern:** `^[a-z0-9]{3,4}[0-9]{3,4}$`

**Structure:** Set code portion followed by a numeric suffix.

- Set code: 3-4 lowercase alphanumeric characters
- Numeric suffix:
  - Values 001-999: zero-padded to exactly 3 digits
  - Values 1000-9999: 4 digits, no padding
  - A 4-digit suffix must be >= 1000. Zero-padded 4-digit suffixes (e.g., `0123`) are never valid.

**Storage:** Always lowercase.

**Display:** Always uppercase in all API responses and UI.

**Valid examples:**

| Stored | Displayed | Notes |
|--------|-----------|-------|
| `eoe019` | `EOE019` | 3-char set code, 3-digit number |
| `eoe999` | `EOE999` | Maximum 3-digit number |
| `eoe1000` | `EOE1000` | Minimum 4-digit number |
| `eoe1234` | `EOE1234` | 4-digit number >= 1000 |
| `3ed019` | `3ED019` | Set code starting with digit |
| `mmh2001` | `MMH2001` | 4-char set code, 4-digit number |

**Invalid examples:**

| Value | Reason |
|-------|--------|
| `eoe0019` | Zero-padded 4-digit suffix; value 0019 < 1000 |
| `Edge of Eternities` | Set name, not a code |
| `EOE019` | Uppercase storage is not accepted; always store lowercase |

## 7. Treatments

Treatments describe the physical variant of a card (foil type, art treatment, etc.). They are never hardcoded - the treatments reference table is received via update packages and all treatment display logic references this table.

Access the current treatments list via `GET /api/treatments` (authenticated). Each treatment has a `key` (stored in lowercase with hyphens), a `displayName`, and a `sortOrder`.

**Current known treatments (as of latest update packages):**

| Key | Display Name |
|-----|-------------|
| `regular` | Regular |
| `foil` | Foil |
| `surge-foil` | Surge Foil |
| `fracture-foil` | Fracture Foil |
| `etched-foil` | Etched Foil |
| `textured-foil` | Textured Foil |
| `galaxy-foil` | Galaxy Foil |
| `serialized` | Serialized |
| `gilded-foil` | Gilded Foil |
| `artist-proof` | Artist Proof |
| `promo` | Promo |

New treatments arrive via update packages without application code changes.

## 8. Condition Values

Condition applies to cards, serialized entries, slabs, and sealed product. It is stored as an enum (not a reference table) and the values are fixed.

| Value | Full Name | Description |
|-------|-----------|-------------|
| `NM` | Near Mint | No visible wear; essentially new |
| `LP` | Lightly Played | Minor surface wear; still clean |
| `MP` | Moderately Played | Visible wear but card is intact |
| `HP` | Heavily Played | Significant wear; noticeable damage |
| `DMG` | Damaged | Major damage; heavily worn or bent |

**Autographed flag:** Autographed is a separate boolean field, not a condition value. When `autographed` is true, the condition is preserved and displayed alongside it. Example display: `NM - Autographed`.

## 9. Universal Filters

Filters are applied across all collection views where contextually appropriate. The `CollectionFilter` model is accepted as query parameters on collection endpoints.

| Filter | Type | Values |
|--------|------|--------|
| `setCode` | string | Set code pattern `^[a-z0-9]{3,4}$` |
| `color` | string | `White`, `Blue`, `Black`, `Red`, `Green`, `Colorless`, `Multicolor` |
| `condition` | string | `NM`, `LP`, `MP`, `HP`, `DMG` |
| `cardType` | string | Top-level MTG types: `Creature`, `Instant`, `Sorcery`, `Land`, `Enchantment`, `Artifact`, `Planeswalker`, `Battle`, etc. |
| `treatment` | string | Treatment key from the reference table |
| `autographed` | boolean | `true` or `false` |
| `serialized` | boolean | `true` or `false` |
| `slabbed` | boolean | `true` or `false` |
| `sealedProduct` | boolean | `true` or `false` |
| `gradingAgency` | string | Agency code |

Filters are applied at the database query level (via EF Core LINQ), not as post-fetch in-memory filtering. A filter that has no effect in a given context is not sent by the UI (e.g., `setCode` is suppressed when already viewing a specific set).

Card type filtering matches on partial string (`Contains`) against the `cardType` field. Subtype filtering is not implemented in the initial release.

## 10. TCGPlayer Price Refresh

Individual card prices can be refreshed from TCGPlayer directly. This overwrites the stored `CurrentMarketValue` for the card immediately. The next content update package from countorsell.com will overwrite this value regardless of which is newer.

**Requires:** A TCGPlayer API key configured by an admin or user. See [docs/configuration.md](configuration.md).

**Endpoints:**
- `POST /api/cards/{identifier}/refresh-price` - refresh and persist the price for a specific card
- `POST /api/collection/refresh-price/{cardIdentifier}` - refresh and return the price (does not persist to the cards table)

When no API key is configured, both endpoints return HTTP 400 with `{"error": "TCGPlayer API key is not configured."}`.

When the TCGPlayer query returns no price, the endpoint returns HTTP 502.
