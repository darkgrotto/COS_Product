# Configuration Reference

---

## 1. Environment Variables

All sensitive configuration is stored as environment variables. The `.env` file (Docker deployments) is generated by the wizard and read by Docker Compose. Cloud deployments inject these values from the provider's secrets manager.

| Variable | Description | Default | Required |
|----------|-------------|---------|----------|
| `POSTGRES_CONNECTION` | PostgreSQL connection string | `Host=localhost;Database=countorsell;Username=countorsell;Password=countorsell` | Yes |
| `INSTANCE_NAME` | Instance branding text shown in header and page title | `CountOrSell` | No |
| `UPDATE_CHECK_TIME` | Time of day for daily content update check (HH:MM, 24-hour, UTC) | None - generated by wizard | Yes |
| `BACKUP_SCHEDULE` | Backup schedule: `weekly`, `daily`, or a cron expression | `weekly` | No |
| `BACKUP_RETENTION` | Number of scheduled backups to retain | `4` | No |
| `BLOB_BACKUP_CONNECTION` | Connection string or URI for the primary backup destination | None | No |
| `OAUTH_GOOGLE_CLIENT_ID` | Google OAuth client ID | None | No |
| `OAUTH_GOOGLE_CLIENT_SECRET` | Google OAuth client secret | None | No |
| `OAUTH_MICROSOFT_CLIENT_ID` | Microsoft OAuth client ID | None | No |
| `OAUTH_MICROSOFT_CLIENT_SECRET` | Microsoft OAuth client secret | None | No |
| `OAUTH_GITHUB_CLIENT_ID` | GitHub OAuth client ID | None | No |
| `OAUTH_GITHUB_CLIENT_SECRET` | GitHub OAuth client secret | None | No |
| `TCGPLAYER_API_KEY` | TCGPlayer API key for direct price queries | None | No |

The application reads OAuth credentials from configuration keys `OAuth:Google:ClientId`, `OAuth:Google:ClientSecret`, `OAuth:Microsoft:ClientId`, `OAuth:Microsoft:ClientSecret`, `OAuth:GitHub:ClientId`, and `OAuth:GitHub:ClientSecret`. These can be set via environment variables using the ASP.NET Core double-underscore convention (e.g., `OAuth__Google__ClientId`).

## 2. Application Settings

These settings are stored in the `app_settings` database table and managed through the admin panel or the `/api/settings` endpoints. They persist across container restarts.

| Key | Description | Default |
|-----|-------------|---------|
| `backup_schedule` | Backup frequency: `weekly` or `daily` | `weekly` |
| `backup_retention_scheduled` | Number of scheduled backups to retain | `4` |
| `backup_retention_pre_update` | Number of pre-update backups to retain | `4` |
| `current_schema_version` | Current applied schema version (managed internally) | `1` |

Backup settings can be read and updated via:
- `GET /api/settings/backup` - returns current values
- `PATCH /api/settings/backup` - update schedule and/or retention counts

## 3. Backup Destinations

One backup destination is configured during the wizard. Additional destinations are added post-setup via `POST /api/backup/destinations`. All active destinations receive every backup. A failure on one destination does not block others - each is attempted independently, and the admin is notified of any failure.

### Destination types

**`local` - Local file export**

Always available. Writes `.zip` files to a local directory path.

Configuration JSON fields:
```json
{
  "path": "/path/to/backup/directory"
}
```

If `path` is omitted, defaults to `<app-base-directory>/backups`.

---

**`azure-blob` - Azure Blob Storage**

Not yet fully implemented - requires Azure.Storage.Blobs SDK integration. The destination type is registered and the wizard can collect a connection string, but write/read/delete operations will throw `NotImplementedException` until the SDK is wired up.

Configuration JSON fields:
```json
{
  "connectionString": "DefaultEndpointsProtocol=https;AccountName=...;AccountKey=...;EndpointSuffix=core.windows.net"
}
```

---

**`aws-s3` - AWS S3**

Not yet fully implemented - requires AWSSDK.S3 SDK integration. Registered as a destination type but write/read/delete operations will throw `NotImplementedException` until the SDK is wired up.

Configuration JSON fields:
```json
{
  "bucket": "my-backup-bucket",
  "region": "us-east-1"
}
```

---

**`gcp-storage` - GCP Cloud Storage**

Not yet fully implemented - requires Google.Cloud.Storage.V1 SDK integration. Registered as a destination type but write/read/delete operations will throw `NotImplementedException` until the SDK is wired up.

Configuration JSON fields:
```json
{
  "bucket": "my-backup-bucket"
}
```

---

### Destination API

| Action | Endpoint |
|--------|---------|
| List destinations | `GET /api/backup/destinations` |
| Add destination | `POST /api/backup/destinations` |
| Remove destination | `DELETE /api/backup/destinations/{id}` |
| Test connection | `POST /api/backup/destinations/{id}/test` |

All destination endpoints require Admin authentication.

`POST /api/backup/destinations` request body:
```json
{
  "destinationType": "local",
  "label": "My Local Backup",
  "configurationJson": "{\"path\":\"/backups\"}"
}
```

## 4. OAuth Provider Configuration

OAuth providers are configured post-setup. Each provider must be registered as an OAuth application in the provider's developer console to obtain a client ID and secret. Providers are only registered with ASP.NET Core authentication if both the client ID and secret are present in configuration.

### Google

1. Go to https://console.cloud.google.com/apis/credentials
2. Create an OAuth 2.0 Client ID (Web application type)
3. Add the callback URL: `https://<your-instance>/api/auth/oauth/google/callback`

Configuration keys:
```
OAuth:Google:ClientId
OAuth:Google:ClientSecret
```

### Microsoft

Supports personal Microsoft accounts (Live) and organizational accounts (Entra ID). The standard Microsoft account OAuth flow handles both.

1. Go to https://portal.azure.com/#view/Microsoft_AAD_RegisteredApps
2. Register a new application
3. Add the redirect URI: `https://<your-instance>/api/auth/oauth/microsoft/callback`

Configuration keys:
```
OAuth:Microsoft:ClientId
OAuth:Microsoft:ClientSecret
```

### GitHub

1. Go to https://github.com/settings/developers and create a new OAuth App
2. Set the callback URL to: `https://<your-instance>/api/auth/oauth/github/callback`

Configuration keys:
```
OAuth:GitHub:ClientId
OAuth:GitHub:ClientSecret
```

## 5. TCGPlayer API Key

The TCGPlayer direct price refresh feature allows querying current market prices for individual cards. This requires an API key from TCGPlayer.

**What it does:** When an admin or user triggers a price refresh for a specific card, the application queries TCGPlayer's API directly and stores the returned price. This price is treated identically to update-package prices in the UI. The next content update package will overwrite the TCGPlayer-sourced price regardless of which is newer.

**Obtaining a key:** Register for API access at https://developer.tcgplayer.com/.

**Configuration:** Set the `TCGPLAYER_API_KEY` environment variable, or configure it via the admin panel settings (stored securely, never in plain text).

**Behavior without a key:** If no API key is configured, the refresh-price endpoints return HTTP 400 with the error `TCGPlayer API key is not configured.` and the feature is unavailable in the UI.

**Direct connection:** CountOrSell connects directly to TCGPlayer from the server. CountOrSell does not proxy this connection through any intermediate service.
