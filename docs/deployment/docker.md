# Docker Compose Deployment

---

## 1. Overview

Docker Compose is the self-hosted deployment option. The wizard generates all required files. The Compose configuration runs four services and requires no Docker socket access from within any container.

| Service | Image | Role |
|---------|-------|------|
| `app` | `${REGISTRY}/countorsell:latest` | CountOrSell application (API + React client) |
| `postgres` | `postgres:16-alpine` | PostgreSQL 16 database |
| `reverse-proxy` | `nginx:alpine` | SSL termination, HTTPS on the configured port |
| `backup` | `${REGISTRY}/countorsell-backup:latest` | Scheduled backup runner |

The `app` and `backup` services depend on the `postgres` service and will not start until PostgreSQL passes its healthcheck (`pg_isready`).

---

## 2. Prerequisites

| Requirement | Notes |
|-------------|-------|
| Docker Desktop (Windows/Mac) or Docker Engine + Compose plugin (Linux) | Must be installed by the user before running the wizard |
| Docker Compose v2 | Included with Docker Desktop; available as a plugin for Linux |
| .NET 8 SDK | Required to run the first-run wizard |
| Available port | Default is 443; configurable during wizard Step 5 |

The wizard detects whether `docker` and `docker compose` are available and blocks until both are present.

## 3. Generated Files

The wizard generates three files during Step 15. **Do not edit these files manually.** Re-run the wizard to regenerate them if changes are needed.

### `docker/compose/docker-compose.yml`

The managed Compose file. Defines all four services, their environment variables, health checks, volumes, and restart policies. Environment variable values are read from `.env` at startup.

Key properties:
- `app` service reads `POSTGRES_CONNECTION`, `INSTANCE_NAME`, `UPDATE_CHECK_TIME`, `BACKUP_SCHEDULE`, `BACKUP_RETENTION`
- `postgres` service uses a named volume `countorsell_postgres_data` for data persistence
- `reverse-proxy` service mounts `docker/compose/nginx.conf` and `docker/certs/` (read-only)
- `backup` service reads `POSTGRES_CONNECTION`, `BACKUP_SCHEDULE`, `BACKUP_RETENTION`, `BLOB_BACKUP_CONNECTION`

### `docker/scripts/update.sh`

The application update script. See Section 5.

### `.env`

Environment variables read by Docker Compose. Contains instance configuration, credentials, registry location, and port. Generated by the wizard and kept in sync by re-running the wizard.

```
INSTANCE_NAME=my-instance
UPDATE_CHECK_TIME=14:37
BACKUP_SCHEDULE=0 2 * * 0
BACKUP_RETENTION=4
DB_USER=countorsell
REGISTRY=registry.example.com
PORT=443
BLOB_BACKUP_CONNECTION=./backups
```

## 4. SSL

A self-signed certificate is generated during wizard Step 6 and placed in `docker/certs/`. The nginx reverse proxy (`reverse-proxy` service) serves HTTPS using this certificate.

The nginx configuration is mounted read-only from `docker/compose/nginx.conf`. It proxies HTTPS requests to the `app` service on port 8080.

For production use, replace the self-signed certificate with a trusted certificate by replacing the files in `docker/certs/` and re-running the wizard to regenerate the Compose file if the certificate paths change.

---

## 5. Updating the Application

When a new application version is available, the admin panel displays a notification with the exact command to run. Navigate to `docker/scripts/` and run:

```bash
./update.sh
```

The script performs:
```bash
docker compose -f docker/compose/docker-compose.yml pull
docker compose -f docker/compose/docker-compose.yml up -d
```

This pulls the latest `countorsell:latest` and `countorsell-backup:latest` images and recreates containers. Schema migrations run automatically on startup if the new version includes them (see Section 6).

No Docker socket access is required from within any container.

---

## 6. Startup Schema Migration

On every container start, the application checks for pending EF Core migrations before accepting traffic. The `StartupMigrationService` runs before all other hosted services.

**Sequence when migrations are pending:**
1. Take a pre-update backup silently
2. If backup fails: abort startup, log error, create admin notification - do not migrate
3. If backup succeeds: run `MigrateAsync()`
4. If migration fails: restore from the pre-update backup automatically, abort startup, notify admin
5. If migration succeeds: continue startup

**When no migrations are pending:** Startup continues normally with no backup.

See [docs/backup-restore.md](../backup-restore.md) for complete details on restore behavior and admin notifications.

---

## 7. Backup

The `backup` service in the Compose file is a dedicated container that runs the scheduled backup job. It connects directly to the `postgres` service via the internal Docker network.

By default, backups are written to the local path configured during wizard Step 11. The `BLOB_BACKUP_CONNECTION` environment variable sets the primary backup destination connection string.

Additional backup destinations can be configured post-setup from the admin panel. See [docs/backup-restore.md](../backup-restore.md) and [docs/configuration.md](../configuration.md).

---

## 8. Data Persistence

PostgreSQL data is stored in the Docker named volume `countorsell_postgres_data`. This volume persists across container restarts and image updates.

Volume name: `countorsell_postgres_data`

To back up or restore the volume data, use CountOrSell's built-in backup system rather than accessing the volume directly. The backup system uses `pg_dump` and `psql` and produces portable, schema-versioned archives.

To completely wipe instance data (destructive): stop the containers and remove the volume with `docker volume rm countorsell_postgres_data`.
